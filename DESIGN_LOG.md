A categorized log of design **decisions** and **changes** — grouped by page → section → component so it's easy to find why something looks the way it does. Newest decisions win. For the deeper "why it works" taste analysis and Simon's references/anti-patterns, see `DESIGN_KNOWLEDGE.md §6`. For token architecture see `DESIGN_SYSTEM.md`.

## 0. Global / Design system (`src/index.css`)
- **Glass grain de-pixelated — softer, cleaner frost (2026-06-08)** — *Simon: the grain on glass read "pixelated".* Cause: `--glass-noise` was a hard-edged per-pixel `feTurbulence` (140px tile, baseFreq 0.8, opacity 0.28) — sharp noise pixels read as blocky static on retina. Fix: added **`feGaussianBlur stdDeviation='0.6'`** after the turbulence (softens hard pixel edges → smooth micro-grain — the key lever), grew the tile **140→200px** + octaves **2→3** (finer, less obvious repeat), opacity **0.28→0.20** (present-but-subtle). Same soft-light blend on `.glass`/`.glass-panel`, filter id `gn`. **Tested quantitatively** (drew each noise to canvas, measured adjacent-pixel difference = a pixelation proxy): blur sweep showed a knee — full-opacity harshness 12.5 raw → ~10 at blur 0.4 (too gentle, was the first attempt) → **4.3 at blur 0.6** → 2.2 at 0.8. Landed 0.6 (smooth, still grainy not cloudy). LIVE on-screen harshness now **0.87** (vs first-ship 1.62, vs original hard ~3.0); deployed token verified end-to-end + glass renders clean (no seams). Knobs: opacity = strength · blur = smoothness · baseFrequency = grain size. *(Separate global film-grain `.grain` (id `n`) over the whole page is untouched — if pixelation persists it's the second candidate.)*
- **Glass cards now frost IMMEDIATELY on scroll-reveal — glass element IS the reveal (2026-06-08)** — *Simon: "blur not appearing immediately on scroll loads."* Investigated the DOM: the prominent glass cards were **wrapped** by the reveal (`<Reveal>`/`<ScrollAnimation>` → glass child), and a wrapper animates **opacity AND transform** — *both* isolate `backdrop-filter` (opacity<1 is a backdrop-root trigger too), so the child's frost was dead for the whole reveal then popped in. Key realization: **`REVEAL_SHIFT` (quick-transform decouple) can't save a wrapped child** — the wrapper's slow opacity fade keeps isolating it. Fix = the glass element must **BE** the reveal (own opacity composites the already-frosted element → frost visible from frame one; own transform is fine). Refactored the four offenders so the glass element animates itself: **ProjectCard** (`<Link>` → module-scope `m.create(Link)` passed as `Reveal as=`), **SkillCard** (`<Reveal className="glass-panel …">`), **TestimonialCard** (`<Reveal as="figure">`), **LogoGrid cell** (`<Reveal>`); extended `<Reveal>` to accept a custom motion component for `as`. Verified in-browser (animations force-finished): all four now `self-animating, wrapped=false`; every visible glass keeps `backdrop-filter` active; build + lint clean; zero console errors across home/archive/about. **Residual (accepted, documented in GLASS §8):** nested glass — the tiny tag pills *inside* project cards still lose frost during the card's ~0.5s reveal (a glass child of an animating glass card can't keep frost — unavoidable); and WorkedAt hover-tooltips read as "wrapped" but are hidden until hover (shown only after the reveal settles → fine). ⚠️ Frost-feel needs Simon's real-browser eyes (preview tab is backgrounded → animations pause).
- **Motion Phase 2b — reassessed: route transitions + desktop dropdown STAY NATIVE (2026-06-08)** — *Investigated the remaining "move to Motion" candidates; the honest engineering call is to leave them native — Motion would add risk to the glass system for no gain.* ① **Route/page transitions** (`.page-enter-nav` + `pageWipe` curtain) — a deliberate, working signature CSS animation (marimba-style seamless same-colour cover→reveal); Motion's strengths (springs/exit/gestures/FLIP) don't improve a curtain sweep, and the wrapper is blur-sensitive (already fixed once via `fill-mode: backwards`). Keep. ② **Header desktop dropdown** — **must stay transform-free**: it's positioned by JS left-edge (not `translateX(-50%)`) *specifically* because any transform on the wrapper kills the glass panel's `backdrop-filter`. Motion animates via transforms → would re-break the dropdown frost. Keep native (it already opacity-animates open+close). **DO NOT convert these to Motion.** ③ **Mobile menu** — the one real gap: `{isMobileMenuOpen && …}` unmounts **abruptly** on close (CSS has enter stagger `menuRowIn`, no exit). Optional AnimatePresence opacity-exit could fix it (glass-safe, contained) — parked pending Simon's call. **Net: the Motion migration is essentially complete after Phase 1 (reveals) + 2a (modal/lightbox exits).** Cross-screen verification (this session, preview capped ~745px + backgrounded so motion-feel needs real-browser eyes): all 9 routes render, **zero console errors**; **no horizontal overflow** at 745 & 375; visible glass keeps `backdrop-filter blur(16/24px)`; **settled Motion reveals reset `transform:none`** (don't kill descendant blur — glass system healthy); footer blur intact; mobile menu opens (27 items). Build + lint clean.
- **Motion Phase 2a — exit animations via AnimatePresence (2026-06-07)** — *The headline reason a library helps: the owned CSS layer couldn't animate elements OUT before unmount (they just vanished or needed manual `setTimeout` hacks).* Converted the two clean overlay cases to **Framer Motion `<AnimatePresence>`**: ① **Lightbox** (`ImageGrid` — About "Pictures") — dropped the manual `closing` state + `setTimeout(260)` choreography; backdrop now `m.div` (opacity in/out, keeps its `blur(8px)`), image `m.img` (scale+fade, keyed by index so it re-plays on prev/next). ② **Modal** (`overlays/Modal`, the ready-but-unwired dialog primitive) — overlay is an `m.div` propagating `open`/`closed` variants to a scrim (fade) + dialog (fade + slight rise/scale), enter **and** exit; all a11y machinery (focus-trap, scroll-lock, Esc, focus-restore) untouched. Removed the now-dead CSS `@keyframes` (lightboxIn/Out/ImgIn/Out, scrim-in, dialog-in) + `.lightboxClosing` so CSS can't fight Motion's inline styles; reduced-motion handled in JS via `useReducedMotion`. Verified: build + lint clean; lightbox **enter** confirmed visually in-browser (full image/arrows/counter/scrim render). ⚠️ **Exit-unmount not time-verifiable here** — the preview tab runs `visibilityState:hidden`, which pauses WAAPI/rAF so AnimatePresence's exit-complete never fires (stuck mounted in-probe); the pattern is canonical (keyed direct child + `exit` prop) and runs in a real visible browser — **Simon to confirm close/reopen feels right**. **Deferred (Phase 2b, blur-sensitive + high-visibility, own focused pass): route/page transitions + Header dropdown/mobile-menu.** Skipped (CSS is right tool): hover/press, count-up; ScrollProgress left exact-tracking by design.
- **Typeface vertically balanced at the source — `@font-face` metric overrides (2026-06-07)** — *Simon: text in buttons/pills/nav wasn't vertically centred — sits high with extra space below; and "why not fix it across ALL uses of the typeface?"* Root cause is the font's metrics, not the components: **BDO Grotesk ships a top-heavy box** — native ascent **800** / descent **177** / cap-height **729** (per 1000 em), so its line-box centre (311 above baseline) sits *below* the caps' optical centre (365), and `align-items:center` therefore parks the letters **~5.5% of font-size too high** everywhere. Chose the **global** fix (over per-component `text-box-trim`): rewrote the metrics on the `@font-face` rule so the box is balanced for **every** use. A box centres caps when **`ascent − descent = cap-height`**, so **`ascent-override: 90.6%`** (906) / **`descent-override: 17.7%`** (177, held at the descender ink so g/j/p/y never clip) / **`line-gap-override: 29.4%`** (keeps `line-height: normal` at its original ~1.377). Verified in-browser: ascent−descent = 729 = cap-height exactly; caps now centre to **~0.1px** in fixed-height buttons/pills (was ~0.8px high at 14px). **Knock-on (inherent to the font-level route):** the balanced box is **1.083em** tall (descender-depth bound), so multi-line display type below that overlaps — raised the sub-floor display line-heights to **1.1**: `.type-display` (desktop 1.0→1.1, sm 1.05→1.1), `.type-display-sm` (lg 1.05→1.1), About `.aboutHeadline` (1.08→1.1). Single-line stat numbers at `line-height:1` left as-is (can't overlap); the testimonial quote glyph is Georgia (unaffected). Verified no real multi-line overlap at 1440 **and** 375 (hero wraps at 1.1, clears the floor); build + lint clean. **The desktop display headings are now a hair more open (1.1 vs the old 1.0–1.05) — a one-line revert if Simon wants them tighter. Simon to eyeball the centring + hero leading in his real browser.**
- **Scroll-reveal → Framer Motion spring (2026-06-07)** — *After hand-tuned CSS easing couldn't land the feel, Simon chose to adopt a motion library specifically for quality; picked the "Calm" spring in the side-by-side lab.* Adopted **Motion (Framer Motion, `motion/react`)** — installed `motion`, wired `<LazyMotion features={domAnimation} strict>` at the App root (features lazy-load as a ~14kb chunk, out of initial paint; use `m.*`). Reveal feel is one source of truth in **`src/components/motion/revealMotion.js`** — Calm spring **stiffness 55 / damping 18 / mass 1**, travel **28px**, stagger **90ms**, presets fade-up/fade/scale-in. **`<Reveal>`** now renders `m[as]` animating `hidden`→`shown`, **`<Stagger>`** injects a per-child `delay` (ms) instead of the old CSS `--i`, and **`<ScrollAnimation>`** just renders `<Reveal>` — so all three share the one spring and can't drift. **Trigger kept on our `useReveal` IntersectionObserver** (rootMargin `-28%`) on purpose — preserves the dialled-in trigger position rather than switching to Motion's `whileInView`. Reduced-motion: starts `shown`, duration 0 (content never depends on motion to be visible). Built a **`/motion-lab`** page (unlinked) with live spring sliders + replay for tuning. Verified in-browser (home): revealed els settle to `opacity 1 / transform none`, below-fold held at `opacity 0 / translateY(28px)`, no console errors; build + lint clean; Motion chunk split (14kb gzip). Best-practices written up in DESIGN_KNOWLEDGE §4.9; decision in §4.11. **Phase 1 = reveals only**; transitions/modal/lightbox/menus/count-up are Phase 2. ⚠️ **OPEN CAVEAT:** the spring animates the `translateY` glide on the *same* (slow) Calm spring, so on glass cards the **frost-pop-in this entry's predecessor fixed may return** (non-zero transform suppresses `backdrop-filter` until the spring settles ~1s). Options if it shows: per-property transition (quick spring/tween on `y`+`scale`, calm spring on `opacity`) — mirrors the prior CSS fix while keeping the slow fade. Awaiting Simon's eyes.
- **Frost no longer "pops in" after the scroll-reveal (2026-06-07)** — *Simon noticed the blur only appeared once the reveal motion finished.* Cause: `backdrop-filter` is suppressed while an element is `transform`ed, and the reveal animates a `translateY` rise on the card wrapper — so for the whole slow reveal the card was a backdrop-root (blur dead), then the frost snapped on when the transform settled. Fix (Simon's pick: keep the glide, shorten the transform): split the reveal so the **positional glide is quick** (new `--motion-reveal-dur-shift: 0.42s`) while **opacity stays slow** (~0.9s). The transform reaches `none` early (frost on) while the card is still mostly faded-out, so the frost is present for the visible part of the reveal instead of popping in at the end. Applied globally on `.fx-reveal` (opacity keeps `--motion-reveal-dur`, transform uses the new shift duration). Verified computed transition = `opacity 0.9s, transform 0.42s`.
- **Cards now read as frosted glass, not solid panels (2026-06-07)** — *Simon: "cards aren't using blur, but the footer is."* Diagnosis (verified): the cards' `backdrop-filter: blur(24px)` WAS active with no blur-breaking ancestor when revealed — but `--glass-bg-strong` was so opaque (dark 0.80 / light 0.66) that over the *smooth* shader the frost was invisible and they read as flat panels, whereas the footer (`--glass-bg` 0.55/0.46) let the shader show through and read "glassy." Fix: lowered `--glass-bg-strong` → **dark 0.62 / light 0.56** so the warm shader frosts through the cards (and dropdowns/tooltips/modal) while keeping enough scrim for text. Verified both themes: card culprit-ancestors = none, blur(24px) active, shader visibly bleeding through. (Real lesson: over a smooth backdrop the "frosted" look comes from translucency + tint + noise, not the blur itself — so panel opacity is the lever.)

- **Scroll-reveal slowed + relaxed (2026-06-07)** — *Simon: every scroll-appear felt "too early and too fast", and the opacity itself revealed too fast (his calm-motion DNA, §6.7/§6.8).* Retuned the central `.fx-reveal` system (covers all `<Reveal>`/`<Stagger>` usage site-wide) rather than per-page: ① **duration `--dur-slow` (450ms) → new dedicated `--dur-reveal: 900ms` (landed on 900 after 450→too fast, 820→too fast, 1200→too slow)** (decoupled from `--dur-slow`, which many other components share); ② **travel** — `--motion-shift-md` 16→**28px** (sm 8→14, lg 24→40) so the glide reads as movement, not just a fade; `fade-up` (opacity+translateY) is the default preset so both channels animate together; ③ **split the easing** — *movement* keeps `--ease-decelerate` (glides in, settles softly) but *opacity* gets its own **`--motion-reveal-ease-opacity: cubic-bezier(0.45,0,0.5,1)`** symmetric ease-in-out so the fade ramps evenly across the full duration instead of front-loading to ~90% in the first third (the decelerate curve was the real cause of "opacity too fast"); ④ **fires much later** — `useReveal` rootMargin `-10%`→**`-28%`** bottom + threshold 0.12→0.1, so it only starts once the element is well into view; ⑤ **stagger relaxed** `--stagger-1..4` 60/120/180/240 → **90/180/270/360ms**.
